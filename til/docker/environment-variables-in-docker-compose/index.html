Environment Variables in Docker Compose
8 Dec 2024 • 2 mins

There are 2 types of environment variables in docker-compose:

  1. Variable substitution in compose files
  2. Container env variables

Both have different functions and are configured differently.

Note: For debugging env variables, use docker-compose config.

## Variable Substitution

These environment variables are passed into compose files with variable substitution

# docker-compose.yml
services:
  portainer:
    image: portainer:latest
    volumes:
        - "${PORTAINER_DATA_DIR}:./data"
    ports:
        - "${PORTAINER_PORT:-9443}:9443"

A default value are provided with :-[VALUE] while mandatory values are provided with ${VAR:?err}.

These env variables are automatically passed with an .env file in the project’s directory.

portainer
  |
  |- docker-compose.yml
  |- .env

$ docker-compose up -d

To specify a different .env file, use the --env-file flag.

portainer
  |
  |- docker-compose.yml
  |- .env
  |- .env.prod

$ docker-compose --env-file .env.prod up -d

### Precedence

  1. Compose file
  2. Shell environment variables
  3. Environment file .env or --env-file
  4. Dockerfile
  5. Not defined

## Container Environment Variables

These are passed into the container for use within the container. These cannot be used for variable substitution directly.

# docker-compose.yml
services:
  portainer:
    image: portainer:latest
	environment:
	  - "INTERNAL=INTERNAL"

$ docker exec -it portainer bash

# inside container
root@server# env
INTERNAL=INTERNAL

To pass a (or multiple) file(s) of env variables, use the env_file: directive.

portainer
  |
  |- docker-compose.yml
  |- portainer.env  

# docker-compose.yml
services:
  portainer:
    image: portainer:latest
	env_file: "portainer.env"
	environment:
	  - "INTERNAL=INTERNAL"

If multiple env files are passed in env_file, the last file in the list takes precedence.

When both environment: and env_file: directives are set, the environment: directive takes precedence for any common variables. This is true even when the values are empty or undefined. To prevent confusion and conflict, always set default values in the env_file.

# docker-compose.yml
services:
  portainer:
    image: portainer:latest
	env_file: "portainer.env"
	environment:
	  - EDITOR=nano
	ports:
	  - "${PORTAINER_PORT:-9443}:9443"

# portainer.env
PORTAINER_PORT=9222
EDITOR=vim
FILE_ONLY=true

$ docker exec -it portainer bash

# inside container
root@server# env
PORTAINER_PORT=9222
EDITOR=nano
FILE_ONLY=true

We observe that

  • PORTAINER_PORT in portainer.env is not used for variable substitution. Instead, it is defined in the container. It is however, useless in the container.
  • The value of EDITOR=vim in portainer.env is overwritten by the EDITOR=nano.
  • FILE_ONLY is defined and not overwritten.
